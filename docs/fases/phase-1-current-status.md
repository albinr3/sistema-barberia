- Ticket de kiosco: cada turno conserva `ticket_number` interno unico con formato `W{yyyyMMddHHmmssfff}` para QR/auditoria/entrada manual y expone `display_ticket_number` diario (`1`, `2`, `3`...) como numero visible en kiosco, impresion, pantalla publica, panel de barbero, autocaja y recibos; Local Admin y reportes administrativos muestran tambien el interno. El boton del kiosko imprime mediante la impresora predeterminada de Windows usando una pagina grafica Windows/GDI optimizada para rollo 80mm con QR real y codigo interno como texto; no envia comandos ESC/POS. Autocaja usa la misma estrategia Windows/GDI para imprimir recibos, reimpresiones y reporte diario en la impresora 80mm predeterminada.
- Administracion local: la cola activa permite cancelar tickets `waiting`, `called` o `in_service`; al cancelar un ticket con barbero asignado, el barbero activo vuelve a `available`, se intenta asignar automaticamente el siguiente ticket `waiting` compatible y se registra auditoria. Local Admin permite reasignar tickets `waiting` o `called` a cualquier barbero activo; si el destino esta disponible, el ticket queda `called`, y si esta ocupado, offline o sin check-in, queda `waiting` reservado para ese barbero usando `requested_barber_ids`. Historial de tickets concluido muestra vista rapida de hoy y busqueda por fechas. El CRUD de Barberos y Servicios esta en paginas separadas en la navegacion izquierda pero utiliza la logica central de operacion local.
- Barber Panel y el scanner de Appointments reutilizan `BarberPanelService.StartService(...)` para tickets walk-in: si un ticket `waiting` o `called` asignado/reservado para otro barbero se escanea desde una estacion `Available` que ya hizo check-in en la rotacion diaria, queda reasignado y pasa a `in_service` con esa estacion; si la estacion destino esta `called` o `in_service`, el ticket queda `waiting` reservado para ese barbero. Cada traspaso automatico registra `ticket.auto_reassigned` in sync para la bitacora final de `/admin/admin-dashboard`. Las citas mantienen la regla estricta de iniciar solo con el barbero de la reserva y desde una estacion con check-in diario.
- Estaciones, comision y rotacion diaria: todo barbero activo requiere `station_number` positivo, visible como `B-#`, unico entre activos; al desactivar se libera la estacion. La cola ya no se controla por un numero manual de `rotation_order` ni por el switch de Barber Public/Admin: Barber Public y Local Admin solo hacen al barbero seleccionable en el kiosko, mientras `BarberRotationPage` crea la rotacion diaria cuando el barbero digita/escanea su estacion. La rotacion conserva la llegada si se apaga y reenciende el mismo dia, prioriza barberos con cero clientes atendidos y autocaja mueve al barbero al final solo si ya tenia entrada de check-in del dia. Cada barbero tiene `commission_percentage` con valor inicial 65%; bases locales existentes reciben 65% al inicializar.
- Reset diario operativo: al primer uso despues de cambiar la fecha operativa de New Jersey/Eastern Time, el sistema cancela tickets activos de dias anteriores, pone todos los barberos offline, limpia `checked_in_at`, reinicia `clients_served_today` y registra auditoria. `barbers.rotation_order` queda como columna legacy sin controlar asignacion.
